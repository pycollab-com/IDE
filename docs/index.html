PyCollab

PyCollab is a browser-based collaborative Python IDE. Users edit the same project in real time, run Python directly in the browser through Pyodide, and share workspaces with a short six-character code.

Platform capabilities

  • Real-time collaborative editing with shared project state.
  • Browser-side Python execution with streamed stdout, stderr, and interactive stdin.
  • Project sharing with six-character codes using digits and lowercase letters.

When to use PyCollab

  • Teaching Python without requiring local environment setup.
  • Pair programming or small-team editing sessions in the browser.
  • Running constrained Python workflows inside a managed runtime.
Execution Pyodide in a dedicated browser worker
Sharing Six-character project codes
Access Guest or Google-backed account access
The backend serves configuration, persistence, files, and collaboration state. User Python code does not run on the server.

Quick start

The product is designed so a new collaborator can go from a blank browser tab to a running shared Python project without local Python tooling.

Open the app

Visit pycollab.com. Users can enter as a guest for fast access or continue with an authenticated account.

Start a project

  1. Create a new project from the dashboard.
  2. Open the starter Python file that is added automatically.
  3. Edit files directly in the browser editor.

Share and join

Generate a project share code and give it to collaborators. The code is six characters long and uses only digits and lowercase letters for fast manual entry.

Run Python

Execute the active Python file. Output streams into the terminal panel in real time, and interactive input() prompts are supported inside the browser runtime. 

print("hello from pycollab")
name = input("Your name: ")
print(f"Hi, {name}")

Architecture

PyCollab is split across a browser UI, a synchronization and persistence layer, and a client-side Python runtime. The runtime boundary is the key product constraint.

System layout

Client shell

Editor and project UI

Project dashboard, files, editor, terminal, sharing flows, and presence indicators.

Collaboration layer

Live shared state

Edits, cursor state, collaborators, and project membership are synchronized in real time.

Persistence layer

Projects and uploads

Projects, files, collaborators, and persisted uploads are stored outside the browser runtime.

Execution layer

Pyodide worker

Python executes in-browser with streamed IO and optional package policy enforcement.

Execution flow

  1. The browser loads project state and collaborative editing data.
  2. Python code is sent to a dedicated Pyodide worker inside the browser.
  3. Stdout, stderr, and stdin messages stream between the worker and terminal UI.
  4. Persisted files and uploads remain backend-managed, but execution stays client-side.

Product guarantees

  • Concurrent edits are merged instead of using last-writer-wins overwrites.
  • Runtime initialization fails clearly when browser isolation requirements are not met.
  • Package loading can be constrained through backend configuration.

Collaboration

Collaboration behavior is intentionally simple: shared files, short invite codes, visible remote activity, and low-friction entry for users who only need to join once.

Sharing model

Share codes are six characters long and resolve directly to a project invitation flow. This keeps the handoff fast in classrooms, demos, and pair-programming sessions.

Editing model

Multiple collaborators can edit the same project concurrently. Remote cursors and selections remain visible so collaborators can coordinate without leaving the editor.

Access model

Guest access

Best for fast entry and temporary sessions where registration would only create friction.

Account access

Required for identity-backed ownership, collaborator management, and Google-verified email flows.

Browser runtime

Python execution is powered by Pyodide. The runtime is configured by the backend but runs in the browser, so correctness depends on browser capabilities and header policy.

Runtime contract

  • The default pinned runtime version is 0.29.3.
  • Execution time limits can be enforced with a max-run-seconds setting.
  • Interactive stdin is supported through streamed worker messages.

Cross-origin isolation

Full-fidelity interrupts and blocking stdin require cross-origin isolation. The browser must expose window.crossOriginIsolated as true. 

Cross-Origin-Opener-Policy: same-origin
Cross-Origin-Embedder-Policy: require-corp

Package policy

Allowed packages can be constrained through a backend allowlist. This governs both runtime imports and micropip.install.

Configuration

The deployment surface is intentionally small. Most runtime behavior is driven through environment variables rather than a large configuration graph.

Local development

Copy the example environment file and set values for your local machine.

cp .env.example .env

Google OAuth

Both the backend and frontend must point to the same Google Web client ID. Email verification is Google-backed only. 

GOOGLE_OAUTH_CLIENT_ID=your_google_web_client_id
VITE_GOOGLE_CLIENT_ID=your_google_web_client_id
